# Инструмент управления игрой (дашборд + админка)

## Быстрый старт (Docker Compose)

1. Запуск:

```bash
docker compose up --build
```

2. Открыть в браузере:
- Дашборд (лидерборд): `http://localhost:8080/`
- Админка:
  - Задачи: `http://localhost:8080/admin/tasks`
  - Раунды: `http://localhost:8080/admin/rounds`

Примечания:
- Миграции БД накатываются автоматически при старте приложения.
- При первом запуске задачи автоматически загружаются из файла `data/tasks_seed.csv` (если ещё не загружены).
- Данные Postgres сохраняются в docker volume `db-data`.
- Чтобы сбросить данные: `docker compose down -v`

# IT‑игра «Ресурсный батл»

Командная настольная игра про баланс ресурсов, приоритизацию и технические компромиссы в IT‑проектах.
## 1. Компоненты
- Набор проектных сценариев (A, B, C, D, E, F) с описанием бизнеса и стартовых условий.
- Карты задач: фичи, техдолг, инфраструктурные и риск‑ориентированные задачи.
- Карты ресурсов команды: разработчики, аналитики, техлидлы, DevOps, дизайнеры, тестировщики.
- Специальные карты (джокеры) — по необходимости.

Точный состав колоды и количество карт фиксируются в листе комплектации конкретной версии игры (например, v2.2).
## 2. Цель игры
- За ограниченное количество раундов набрать как можно больше очков за реализованные задачи и удовлетворённость стейкхолдеров.
- Не «сломать» систему: не накопить критический техдолг, не выйти за ресурсные ограничения и SLA.
## 3. Роли и команды
- 3–6 команды по 3–6 человек
- В каждой команде выбирается координатор, который озвучивает решения и следит за временем.
- Ведущий (фасилитатор) объясняет правила, следит за раундами и разрешает спорные ситуации.
## 4. Подготовка к игре
1. Каждая команда получает проект (A/B/C/D/E/F).
2. Ведущий объявляет:
   - количество раундов (обычно 3–5),
   - длительность раунда (обычно 15 минут),
   - используемые доп. правила (джокеры, ивенты и т.п.).
## 5. Структура раунда

Каждый раунд состоит из четырёх фаз.
### 5.1. Планирование (до 5 минут)
- Команды выбирают, какие задачи брать в работу в текущем раунде.
- Учитываются доступные ресурсы, риски, приоритеты бизнеса, ограничения по SLA и техдолгу.
- Команда фиксирует решения: какие задачи делает, какие откладывает или «режет».
### 5.2. Переговоры (до 10 минут)
- Менеджеры от команды идут за стол переговоров и торгуются за ресурсы
- Каждый участник команды может быть менеджером 1 раз
- На этом этапе выдаются джокеры
### 5.3. Исполднение (до 5 минут)
- Команлы борются с джокерами
- Команды исполняют задачи
### 5.4. Подсчёт очков (1–2 минуты)
- За каждую полностью выполненную задачу начисляются очки по шкале сценария.
- За нарушения ограничений (перерасход ресурсов, игнор SLA, невыполненные обещанные задачи и т.п.) начисляются штрафы.
## 6. Ресурсы и задачи

- Каждый ресурс (роль) имеет ограниченное количество поинтов на раунд.
- Задача считается выполненной, если закрыты все требуемые по ней ресурсы (аналитика, разработка, тестировщики и т.д.).
- Если команда пообещала задачу в этом раунде, но не закрыла её ресурсами:
  - задача может не приносить очки,
  - команда получает штраф (механика описана в сценарии).
- Техдолг и риски копятся от раунда к раунду; при достижении пороговых значений запускаются дополнительные штрафы или ограничения (например, потеря части ресурсов, дополнительные обязательные задачи и т.п.).

Конкретные цифры по ресурсам, стоимости задач, бонусам и штрафам определяются сценарием (лист сценария для версии v2.2 или выше).
## 7. Подсчёт очков и победа

В конце последнего раунда:

- Суммируются:
  - очки за выполненные задачи,
  - штрафы за техдолг, риски, перерасход ресурсов и проваленные обязательства.
- Побеждает команда с максимальным итоговым количеством очков.
- При равенстве очков побеждает команда с меньшим объёмом техдолга и штрафов (по правилам сценария).
## 8. Рекомендации ведущему

- Провести тренировочный «нулевой» раунд без жёсткого подсчёта очков, чтобы участники почувствовали механику.
- В обсуждении акцентировать не только результат (очки), но и ход мыслей команд: как они балансировали фичи, техдолг и риски.
- В финале сделать короткий разбор:
  - какие паттерны решений повторялись у разных команд,
  - где команды «продавались бизнесу»,
  - где недооценивали эксплуатацию или техдолг.
## 9. Версии и кастомизация

- Эта версия правил описывает общую структуру игры и ориентирована на редакцию сценариев v2.2.
- Для каждого конкретного мероприятия ведущий может:
  - выбрать подмножество сценариев,
  - упростить или усложнить механику штрафов и джокеров,
  - настроить длительность раундов и количество раундов под слот сессии.
